# Implementation Plan: Web Pages тАФ Initial Layout & Auth Forms

**Branch**: `005-web-pages-layout` | **Date**: 2026-03-21 | **Spec**: [spec.md](./spec.md)

## Summary

Build the first set of public-facing pages for Plotter (home, login, sign-up, reset-password, dashboard placeholder) along with an Axios API client singleton, typed auth endpoint wrappers, Zustand auth store, three prop-driven form components with matching hooks, and an auth-aware navigation bar тАФ all wired through TanStack Router file-based routes and TanStack Query mutations.

**Language/Version**: TypeScript 5.9, React 19

**Project Type**: Web application (frontend)

**Constraints**: Base URL must be environment-configurable via `VITE_API_BASE_URL`; `withCredentials: true` on the Axios instance for cross-origin cookie sessions during local dev  
**Scale/Scope**: 5 pages, 1 layout component, 3 form components, 3 hooks, 1 API module, 1 Axios singleton, 1 Zustand store

## Design Decisions

### Axios Singleton (`web/src/lib/apiClient.ts`)

```ts
import axios from "axios";

export const apiClient = axios.create({
  baseURL: import.meta.env.VITE_API_BASE_URL,
  withCredentials: true, // required for session cookie on cross-origin dev
  headers: { "Content-Type": "application/json" },
});
```

- `VITE_API_BASE_URL` resolves at Vite compile time from `.env.development` / `.env.production`.
- `withCredentials: true` ensures `Set-Cookie` from the Express session flows correctly when the Vite dev server runs on a different port from Express.
- All API functions import `apiClient` тАФ never `axios` directly.

### `ApiError` class (`web/src/api/types.ts`)

```ts
export class ApiError extends Error {
  constructor(
    public readonly status: number,
    public readonly serverMessage: string,
  ) {
    super(serverMessage);
    this.name = "ApiError";
  }
}
```

API functions catch Axios errors and re-throw as `ApiError`. Hooks map `ApiError.status` to user-facing strings:

| Status    | User-facing message                                               |
| --------- | ----------------------------------------------------------------- |
| 400       | Show `serverMessage` inline (already human-readable from backend) |
| 401       | "Invalid email or password."                                      |
| 409       | "An account with this email already exists."                      |
| 429       | "Too many attempts. Please try again later."                      |
| 0 / other | "Something went wrong. Please try again."                         |

### Form Hook Pattern

Each hook:

1. Manages `fields` (string values), `touched` (boolean per field), `fieldErrors` (string | undefined per field).
2. Validates on blur тАФ marks touched + runs validation; does not show errors for untouched fields.
3. Validates all fields on submit attempt тАФ marks all fields touched.
4. Uses `useMutation` from TanStack Query with the typed API function.
5. Returns a flat props object consumed by the form component.
6. On `mutate` success: calls the provided `onSuccess` callback.
7. On `mutate` error: catches `ApiError`, sets `formError` string for display in an `<Alert>`.

### Root Layout & Auth Hydration

`__root.tsx` runs `useQuery({ queryKey: ['me'], queryFn: getMe, retry: false })`. On success it calls `authStore.setUser(data.user)`; on error it calls `authStore.clearUser()`. The root renders only `<Outlet />` тАФ **no visual chrome**. This keeps the three layout zones cleanly separated.

### Three Layout Zones

| Zone      | Route file                                                                               | Visual chrome                                                     |
| --------- | ---------------------------------------------------------------------------------------- | ----------------------------------------------------------------- |
| Home      | `routes/index.tsx` тЖТ `pages/home.tsx` (renders `<Topbar>` + content)                     | Topbar with "plotter" wordmark + auth-aware links                 |
| Auth      | `routes/_auth.tsx` (pathless layout) + `routes/_auth/{login,sign-up,reset-password}.tsx` | Full-height centered layout, Flowbite `<Card>` wrapper, no Topbar |
| Dashboard | `routes/dashboard.tsx` (layout) + `routes/dashboard/index.tsx`                           | Pass-through `<Outlet>` only; future sidebar/topbar added here    |

- `Topbar.tsx` is a standalone presentational component used by `pages/home.tsx`. It reads `authStore` to show authenticated vs. unauthenticated links.
- `_auth.tsx` renders a full-viewport centered `<div>` with a Flowbite `<Card>` containing `<Outlet />`. Auth page components supply only the form content.
- `dashboard.tsx` renders `<Outlet />` with no wrapper тАФ the dashboard shell is intentionally deferred.

### Password Validation (frontend, aligned with backend)

| Rule                         | Check                        |
| ---------------------------- | ---------------------------- |
| Minimum 12 characters        | `value.length >= 12`         |
| Maximum 128 characters       | `value.length <= 128`        |
| At least 1 uppercase         | `/[A-Z]/.test(value)`        |
| At least 1 special character | `/[^a-zA-Z0-9]/.test(value)` |

> **Note**: The spec draft said "5+ characters". The backend enforces 12-128 characters. Frontend rules are aligned to the backend. The spec should be amended.
